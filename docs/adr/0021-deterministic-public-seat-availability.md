# ADR 0021 - 공개 좌석 빈자리 조회의 결정적 도구 선택

| 항목 | 내용 |
|---|---|
| 날짜 | 2026-07-16 |
| 상태 | Accepted |
| 범위 | 도서관 agent의 층별 공개 좌석 조회 |
| 관련 | [ADR 0012](0012-supervisor-narration-suppression.md), [ADR 0020](0020-personal-academic-auth-gating.md) |

## 배경과 운영 재현

도서관 층별 좌석 집계를 비로그인 공개 도구로 전환하고 대화 턴 격리 수정까지 배포한 뒤,
운영에서 사용자 문장 그대로 `도서관 5층 빈 자리 있어?`를 새 익명 스레드에 보냈다. 로그인
요구와 과거 턴 오염은 사라졌지만 library agent는 실시간 `get_library_seat_status`가 아니라
정적 `get_library_seat_catalog`를 두 번 호출했다. 최종 답변은 좌석 종류와 배치만 근거로
"빈 자리가 있습니다"라고 단정했다.

같은 시점의 ssuMCP 공개 REST와 MCP 직접 호출은 5층의 실제 이용 가능 좌석 수를 정상 반환했다.
따라서 provider나 공개 인증 경계의 문제가 아니라 agent의 도구 선택 문제였다. 이번에는 우연히
실제 빈 좌석이 있었지만, 만석일 때도 카탈로그의 좌석 존재를 빈자리로 오해할 수 있어 사용자
의사결정에 잘못된 실시간 정보를 줄 위험이 있었다.

## 기대와 실제, 영향

- 기대: 명시된 2·5·6층의 `빈자리`, `좌석 현황`, `자리 있어?` 요청은 공개 실시간 집계 도구를
  정확히 한 번 호출하고 그 수치만으로 답한다.
- 실제: prompt가 공개 도구 사용을 요구했지만 모델이 유사한 이름의 정적 카탈로그를 선택했다.
- 영향: 로그인 UX는 고쳐졌어도 답변의 데이터 출처가 틀렸다. 정적 배치 정보는 실시간 가용성을
  증명하지 못한다.

## 근본 원인과 증거

library agent는 모든 공개 조회를 같은 ReAct 도구 선택 단계에 맡겼다. system prompt에는
빈자리 질문에서 공개 읽기 도구를 쓰라는 문장이 있었지만 `get_library_seat_status`를 코드로
강제하지 않았다. 두 도구 모두 이름과 설명에 `seat`가 들어가므로 모델은
`get_library_seat_catalog`도 그럴듯한 선택으로 볼 수 있었다.

기존 회귀 테스트도 모델이 처음부터 `get_library_seat_status`를 선택하도록 scripted response를
주입한 뒤 로그인 문구가 없는지만 확인했다. 테스트가 도구 선택 규칙을 검증한 것이 아니라
원하는 선택을 fixture에 미리 써 놓은 셈이라 운영 오선택을 잡지 못했다.

## 결정

명시 층 하나와 순수한 전체 빈자리 의도가 함께 있는 공개 집계 요청만 LLM 이전에 분류한다.
부분 문자열을 찾고 제외어를 늘리는 방식 대신, 허용하는 전체 발화 문법을 `fullmatch`한다. 문법은
선택적인 도서관 명칭, 숫자 층 하나, `빈자리 있어?`·`몇 석 남았어?`·`좌석 현황 알려줘` 같은
전체 집계 표현과 종결 표현만 받는다. 지원 층은 ssuMCP 계약과 같은 2·5·6층으로 제한한다.
일반 `좌석이 몇 개야?`는 전체 수용량 질문일 수 있으므로 이 분기에서 제외한다. `빈자리`·
`이용 가능 좌석`이 명시되거나 일반 좌석에 `남았어`·`비어 있어`가 붙은 경우만 현재 가용 수로
해석한다.

예약·신청·배정·선점·확보·예매 표현이 함께 있으면 이 분기를 타지 않고 기존 인증·HITL 예약
경로가 우선한다. 전체 문법에 맞지 않는 복수·범위·부정 층, 콘센트·창가 같은 좌석 속성,
추천·배치·위치, 이용시간, 비교·복합 질문도 전체 집계로 축약하지 않고 기존 ReAct 경로에 남긴다.

분류된 요청은 `get_library_seat_status(floor, compact=false)`를 코드에서 직접 호출한다. 응답의
요청 층과 비음수 `availableSeats`를 필수로 검증하고, 운영 좌석·사용 중·예약·사용 불가 수와
구역별 가용 수를 결정적 문장으로 포맷한다. 모델은 도구 선택에도, 수치 해석에도 관여하지 않는다.
응답이 malformed이거나 도구가 실패하면 카탈로그나 LLM 추측으로 폴백하지 않고 잠시 후 다시
시도하라는 고정 문구를 반환한다. 예외 원문은 사용자·checkpoint에 남기지 않고 타입만 로그한다.

## 검토한 대안

### Prompt와 도구 설명만 강화

모델·provider가 바뀌면 다시 어길 수 있고 이미 같은 방식이 운영에서 실패했다. 코드로 판정할 수
있는 좁은 고빈도 의도에 확률적 선택을 유지할 이유가 없다.

### 카탈로그 도구를 library agent에서 제거

좌석 배치, 콘센트·창가 같은 정적 특성 질문과 예약 추천에는 여전히 필요하다. 도구 자체를
없애면 정상 기능을 줄이고 두 데이터의 의미 차이도 해결하지 못한다.

### 상태 도구 호출 뒤 LLM이 최종 문장을 합성

도구 선택 오류는 막지만 수치를 잘못 읽거나 과장할 여지가 남고 추가 모델 비용과 지연이 든다.
층별 aggregate는 schema가 작고 표현도 정형적이므로 결정적 포맷이 더 신뢰할 수 있다.

## 검증과 재발 방지

- scripted LLM이 정적 카탈로그 답을 준비해도 정확한 빈자리 문장은 LLM을 한 번도 호출하지
  않고 실시간 상태 도구 결과로 생성한다.
- 2·5·6층의 표현과 띄어쓰기 변형을 분류하고 예약 의도는 무세션 인증 게이트 또는 연결된
  prepare/HITL 경로가 우선한다.
- 복수·부정 층, 시간·속성·배치·비교 질문은 결정적 aggregate 분기에 들어가지 않는다.
- 응답 층 불일치, 누락·음수·잘못된 타입, 도구 예외는 추측 없이 고정 일시 실패 안내로 끝난다.
- 실패 안내에는 로그인, 내부 세션 값, provider 응답 원문이 없다.
- 운영에서는 SSE tool event가 `get_library_seat_status`인지, 최종 숫자가 같은 시점의 공개 MCP
  응답과 일치하는지 확인한다.

## 남은 위험과 면접 질문

층을 생략한 "도서관 빈자리 있어?"는 어느 층을 합칠지 제품 결정이 필요해 기존 LLM 흐름을
유지한다. 향후 전체 층 aggregate 계약을 추가하면 같은 결정적 분기에 포함한다. 학교 upstream의
집계 자체가 지연될 가능성은 남지만, 정적 카탈로그로 현재 상태를 꾸미지는 않는다.

- 왜 prompt 강화보다 일부 의도의 결정적 dispatch를 선택했는가?
- 정적 카탈로그와 실시간 상태를 같은 도구 집합에 둘 때 의미 혼동을 어떻게 막는가?
- 외부 상태 조회 실패 시 왜 다른 데이터로 그럴듯하게 답하지 않고 fail-closed 하는가?
